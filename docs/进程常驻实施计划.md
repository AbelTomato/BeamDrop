# 进程常驻实施计划

## 目标

将 `beamdrop serve` 从“接收一个传输会话后退出”调整为“启动后持续监听并连续接收多个传输会话”。

当前接收端每次只执行一次 `TcpServer::accept_one()`，完成一个 `HELLO -> FILE_INFO... -> FINISH` 会话后返回 `0`。这会导致接收端在一次发送完成后退出，用户需要手动重新启动服务端。

## 范围

本计划只覆盖接收端进程常驻能力，不改变现有传输协议、不引入并发接收、不改变发送端命令行行为。

涉及模块：

- `src/main.cpp`：`serve` 命令入口、单连接会话处理、常驻循环。
- `src/network/TcpServer.cpp`：现有监听 socket 与 `accept_one()` 行为确认，不预期修改。
- `src/transfer/Receiver.cpp`：现有单会话文件接收逻辑复用，不预期修改协议流程。
- `tests/integration/`：新增或扩展连续接收集成测试。
- `README.md`：更新 `serve` 使用说明。

## 现状结论

`src/main.cpp` 的 `run_serve()` 当前流程：

1. 解析 `serve` 参数。
2. 创建 `TcpServer`。
3. 调用一次 `server.accept_one()`。
4. 读取并校验 `HELLO` manifest。
5. 构造 `Receiver` 接收 `file_count` 个文件。
6. 读取并校验 `FINISH`。
7. 输出完成信息并 `return 0`。

`TcpServer::accept_one()` 本身没有一次性限制。只要 `TcpServer` 对象仍然存活，监听 socket 可继续接受后续连接。因此常驻机制主要应在 `run_serve()` 层实现。

## 设计决策

### 默认常驻

`beamdrop serve` 默认常驻。用户启动接收端后，进程持续监听端口，连续处理多个发送端连接。

退出方式暂定为 Ctrl+C 或外部终止进程。MVP 阶段不额外实现优雅停机命令。

### 串行处理连接

服务端一次只处理一个连接。当前 `Receiver`、续传状态文件和接收目录都没有并发写保护，直接引入多线程接收会增加状态竞争风险。

串行策略：

1. `accept_one()` 等待一个连接。
2. 完整处理该连接的一个传输会话。
3. 关闭连接对象生命周期。
4. 回到 `accept_one()` 等待下一个连接。

### 单连接失败不终止服务

单个客户端连接出现协议错误、连接中断、校验失败或写文件失败时，服务端记录错误并继续监听下一个连接。

绑定失败、监听失败等启动阶段错误仍然应导致 `serve` 退出，因为服务没有成功启动。

## 实现切片

### 切片 1：抽取单连接会话处理

修改 `src/main.cpp`。

新增内部函数，建议形态：

```cpp
std::uint64_t receive_transfer_session(const beamdrop::network::TcpConnection& connection,
                                       const ServeOptions& options,
                                       const beamdrop::logger::Logger& logger);
```

职责：

- 读取 `HELLO` packet。
- 校验 packet type 为 `PacketType::Hello`。
- 解码 `TransferManifest`。
- 构造 `Receiver`。
- 调用 `receiver.receive_files(options.save_dir, file_count)`。
- 读取并校验 `FINISH`。
- 返回本次会话接收的文件数量。

验收：

- 原有单次传输行为保持不变。
- 协议错误仍抛出异常，由调用层决定是否继续监听。

### 切片 2：将 `run_serve()` 改为常驻循环

修改 `src/main.cpp`。

建议流程：

```cpp
beamdrop::network::TcpServer server{options.host, options.port};
std::cout << "beamdrop serve listening ..." << '\n';

while (true) {
    try {
        auto connection = server.accept_one();
        logger.info("serve accepted connection");
        const auto file_count = receive_transfer_session(connection, options, logger);
        std::cout << "beamdrop serve received " << file_count << " file(s)" << '\n';
        logger.info("serve completed file_count=" + std::to_string(file_count));
    } catch (const std::exception& error) {
        logger.error(std::string{"serve connection failed: "} + error.what());
        std::cerr << "beamdrop serve connection error: " << error.what() << '\n';
    }
}
```

注意：

- `TcpServer` 构造应保持在循环外。
- 启动阶段异常仍由外层 `run_serve()` catch 记录为 `serve failed` 并向上抛出。
- 连接处理异常只影响当前连接。

验收：

- 第一次发送完成后，接收端进程不退出。
- 第二次发送可以复用同一接收端进程。
- 单个坏连接不会导致接收端退出。

### 切片 3：连续接收集成测试

新增集成测试文件，建议：

- `tests/integration/persistent_serve_tests.cpp`

测试目标：验证同一个监听服务可连续处理两个完整传输会话。

测试策略：

1. 创建临时目录和两个源文件。
2. 启动服务线程。
3. 服务线程创建 `TcpServer`，循环处理 2 次 `accept_one()`。
4. 每次连接均按完整 CLI 协议处理：读取 `HELLO`、调用 `Receiver::receive_files()`、读取 `FINISH`。
5. 主线程创建两个 `TcpClient`，分别发送一个文件，且每次都写入 `HELLO` 和 `FINISH`。
6. 断言两个目标文件均存在且内容正确。

该测试不应直接调用无限循环后的 `run_serve()`，否则测试无法自然结束。应测试同一 `TcpServer` 连续接受多个连接的核心能力，或在实现中抽出可调用的单连接处理函数后测试该函数。

单元测试任务：

- 无新增纯单元测试要求；本改动主要是命令入口控制流，集成测试覆盖更有效。

集成测试任务：

- 新增 `persistent_serve_tests`。
- 将测试加入 `CMakeLists.txt` 的测试目标。
- 运行 `ctest --test-dir build-mingw --output-on-failure`。

## 验证命令

构建：

```bash
cmake --build build-mingw
```

运行测试：

```bash
ctest --test-dir build-mingw --output-on-failure
```

手工验证：

```bash
build-mingw\beamdrop.exe serve --config config\beamdrop.example.json --host 127.0.0.1 --port 9090 --save-dir received
```

在另一个终端连续执行：

```bash
build-mingw\beamdrop.exe send README.md --to 127.0.0.1:9090
build-mingw\beamdrop.exe send docs --to 127.0.0.1:9090
```

预期：

- `serve` 在第一次发送完成后不退出。
- 第二次发送成功。
- `received/` 中存在两次发送的结果。

## 成功标准

- `beamdrop serve` 默认常驻，能连续处理多个发送连接。
- 单连接传输失败不会终止服务端进程。
- 现有单文件、目录、断点续传测试继续通过。
- 新增连续接收集成测试通过。
- `README.md` 说明与实际行为一致。

## 非目标

- 不实现多客户端并发接收。
- 不实现服务端优雅停机命令。
- 不修改 packet 协议格式。
- 不修改 `send` 命令行为。
- 不引入后台守护进程、系统服务或托盘进程。
