## 接收服务 stop 卡在 `stopping` 修复

### 根因与修复

- `/api/receiver/stop` 返回 `200` 仅表示停止请求已受理；原生 `ReceiveServerService::stop()` 是异步状态转换，真正退出前状态仍为 `stopping`。
- 接收线程在已建立的连接上阻塞读取 `HELLO` 或传输数据时，旧实现没有把 worker 的停止请求传递到 socket，因此只能等待对端完成、断开或网络超时。
- `TcpConnection` 现在以原子句柄管理 socket，并新增线程安全 `shutdown()`：关闭 socket 以解除其他线程中的阻塞读写。`ReceiveServerService` 为每个已接入会话注册 `std::stop_callback`，`stop()` 请求抵达即中断该会话，接收线程随后收敛为 `stopped`。
- 该改变的语义是**立即停止接收服务并中断当前接收会话**；未完成文件保留既有续传状态，由下次传输恢复。

### 验证

- 新增集成场景：发送端仅建立 TCP 连接、不发送 `HELLO`，然后调用 `stop()`；断言服务在 2 秒内从 `stopping` 进入 `stopped`。
- `ctest --preset windows-mingw-debug --output-on-failure`：3/3 通过。
- `.venv\Scripts\python.exe -m pytest backend\tests -q`：29/29 通过，保留 1 条既有 Starlette/httpx 弃用警告。

## GUI 手工验收阻塞修复

### 根因

- `FastApiAdapter` 将浏览器原生 `fetch` 保存为未绑定函数后，以 `this.request(...)` 调用。浏览器要求 `fetch` 的 receiver 为 `window/globalThis`，实际 receiver 被替换为 adapter 实例，因此启动时抛出 `TypeError: Illegal invocation`。
- adapter 将该异常统一转换为 `BACKEND_UNAVAILABLE`，造成后端健康、WebSocket 已连接且浏览器手工 fetch 正常时，GUI 仍错误显示“后端离线”。
- React StrictMode 会立即关闭首次 effect 中创建的 WebSocket；后端未捕获发送阶段的断连异常，Windows Proactor 因而输出 `WinError 10054` 和重复 `socket.send()` 日志。

### 修复与验证

- 默认 fetch 改为 `globalThis.fetch.bind(globalThis)`；保留可注入 fetch 的 adapter 测试边界。
- 新增回归测试，断言默认 fetch 以 `globalThis` 为 receiver 调用。
- `/ws/events` 发送循环捕获 `WebSocketDisconnect`、`OSError` 与 `RuntimeError`，正常退出并注销订阅。

## GUI 链路复核与取消语义修复

### 复核结论

- 已重新审阅完整运行路径：React feature/store → `BeamDropApi` → `FastApiAdapter` → REST/WebSocket → `TaskManager`/`ReceiverManager` → `PybindNativeGateway` → pybind11 → C++ app service → TCP/transfer core。
- REST snapshot 仍是权威状态；WebSocket 仅投递增量。连接建立、重连和 sequence 缺口均会触发 snapshot 刷新。`EventBus` 使用订阅者独立有界队列，慢客户端可丢弃非终态进度，不能反压 native 传输线程。
- 异常映射保持结构化：C++ `ServiceError` → pybind `BeamDropError(code)` → `NativeGatewayError` → task `failed`/`canceled` snapshot → REST/WebSocket `ErrorPayload`。网络、文件系统、协议与取消均不依赖解析日志文本。

### 修复内容

- 修复取消竞态：此前任务进入 `canceling` 后，如果 C++ native 实际正常返回，Python 层会错误改写为 `canceled`。取消是协作式请求，不是传输未完成的证据；现在仅 C++ 明确返回 `CANCELLED` 时进入 `canceled`，native 成功返回统一进入 `completed`。
- 修复 `PybindNativeGateway` 每次发送创建的取消 watcher：正常完成时 watcher 原先永久阻塞在 `cancel_event.wait()`。现在由完成事件唤醒并 `join()`，不会随成功任务累计 daemon 线程。
- 收紧 `FastApiAdapter` 事件解析：移除宽泛的 `task.*` / `receiver.*` 前缀解析，改为精确已知事件集合；未来 `task.paused` 等未知事件保留原始 payload，由页面安全忽略，不会错误按 task DTO 解析并触发 `INVALID_BACKEND_RESPONSE`。

### 验证

- 新增取消请求与 native 成功完成竞态测试，断言最终状态为 `completed`。
- 新增未知 task-like WebSocket 事件兼容性测试。
- `.venv\Scripts\python.exe -m pytest backend\tests -q`：30/30 通过，保留 1 条既有 Starlette/httpx 弃用警告。

### 已知边界

## 接收服务停止状态未收敛修复

### 根因

- GUI 的 `POST /api/receiver/stop` 已成功到达 FastAPI，但 C++ `ReceiveServerService::stop()` 的返回值表达的是“已请求停止”，正常情况下为 `stopping`，而不是 worker 已实际退出后的 `stopped`。
- `ReceiverManager.stop()` 原先直接把该临时 native snapshot 作为 REST 响应和 `receiver.stopped` WebSocket 事件发布。C++ worker 随后虽会自行把内部状态设为 `stopped`，但没有回调通知 Python manager，因此前端持有的最后状态永久停在 `stopping`，停止按钮被禁用。

### 修复与验证

- `ReceiverManager.stop()` 仍在 FastAPI worker thread 中执行；取得 native 的 `stopping` 后每 25 ms 查询一次 `receiver_status()`，最多等待 2 秒。
- 仅在 native 状态实际收敛为 `stopped` 后，才写入 manager snapshot 并发布 `receiver.stopped`；native 返回失败或超时改为明确 `failed`（`STOP_TIMEOUT`），不再将未完成停止伪装为 stopped。
- 新增 manager 回归测试，模拟 native 先返回 `stopping`、后续 status 才返回 `stopped` 的正常异步停止流程。
- `.venv\Scripts\python.exe -m pytest backend\tests -q`：31/31 通过；`npm --prefix frontend test`：10/10 通过；`npm --prefix frontend run build`：通过；Windows CTest：3/3 通过。

### 手工验证要求

- 该修复需要重启当前正在运行的 `backend/run_windows_dev.py` 才会加载。重启后执行“启动接收服务 → 等待任意时间 → 停止接收服务”，REST 响应、WebSocket 事件和 GUI 最终状态应为 `stopped`；若 2 秒内未收敛，GUI 应显示 `failed` 及 `STOP_TIMEOUT`，而不会无限卡在 `stopping`。
