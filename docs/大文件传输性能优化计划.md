# 大文件传输性能优化计划

## 背景问题

2026-07-03 进行大文件传输测试时，观察到以下现象：

- 传输了一个约 `5.9GB` 的文件，总耗时约 `8 分半`。
- 两端使用 `iperf3` 测试后，链路吞吐约 `340MB/s`。
- 开始正式传输前存在较长停顿。
- 断点续传时，续传间隔体感偏久。

粗略换算：`5.9GB / 510s ≈ 11.6MB/s`，明显低于 `iperf3` 测得的链路吞吐。因此当前瓶颈不应优先归因于网络带宽，而应优先检查应用层传输流程、磁盘 I/O、校验逻辑、状态落盘和控制台输出。

## 已确认代码路径

### 发送前完整 SHA-256 计算导致启动停顿

当前 `src/transfer/Sender.cpp` 在发送 `FILE_INFO` 前会同步计算整个源文件的 SHA-256：

```cpp
const auto file_size = std::filesystem::file_size(source_path);
const auto file_hash = utils::sha256_file(source_path, chunk_size_);

protocol::Packet info_packet;
info_packet.header.type = protocol::PacketType::FileInfo;
info_packet.payload = FileInfoCodec::encode(FileInfo{relative_path, file_size, file_hash});
protocol::write_packet(connection_, info_packet);
```

这意味着大文件在网络传输开始前会被完整读取一遍。接收端在此阶段只能等待 `FILE_INFO`，表现为“开始传输前长时间停顿”。

结论：该停顿不是写入缓冲区时间过长导致的直接现象，而是发送端预哈希导致的同步阻塞。

### 接收完成后再次整文件校验增加总耗时

当前 `src/transfer/Receiver.cpp` 在文件接收完成后会再次读取输出文件计算 SHA-256：

```cpp
if (utils::sha256_file(output_path, 1024 * 1024) != file_info.sha256) {
    throw std::runtime_error("received data sha256 mismatch");
}
```

因此一次完整传输至少包含：

1. 发送端预读完整源文件计算 SHA-256；
2. 发送端再次读取源文件进行网络发送；
3. 接收端写入目标文件；
4. 接收端再次读取目标文件计算 SHA-256。

`iperf3` 只测链路吞吐，不包含上述文件扫描、校验和应用层状态维护成本。

### 断点续传状态每个 Data 包落盘

当前 `src/transfer/Receiver.cpp` 每收到一个 `Data` 包就更新断点续传状态：

```cpp
if (enable_resume_) {
    resume_manager.update_offset(file_info.relative_path,
                                 file_info.size,
                                 file_info.sha256,
                                 received_size);
}
```

而 `src/transfer/ResumeManager.cpp` 的 `update_offset()` 每次都会：

1. `load()` 读取并解析状态 JSON；
2. 修改或新增记录；
3. `save()` 整体重写状态 JSON。

默认 chunk size 为 `1MiB` 时，`5.9GB` 文件约产生 `6000` 个 Data 包，即约 `6000` 次状态文件读写。该行为会显著拖慢大文件传输，并导致断点续传相关操作间隔偏久。

### Data 包存在额外内存复制

当前发送端每个 chunk 会经历：

1. `ifstream.read()` 读入 `buffer`；
2. `data_packet.payload.assign(...)` 复制到 payload；
3. `Serializer::encode(packet)` 再复制到完整 packet vector；
4. `TcpConnection::write_all(...)` 写入 socket。

该路径不是当前最明确的首要瓶颈，但在高速局域网、大 chunk、大文件场景下会增加 CPU 和内存带宽开销。

### 待确认因素：进度输出频率

当前传输循环每个 chunk 都会触发进度回调。若 `main.cpp` 中的进度输出每次都刷新控制台，Windows 控制台 I/O 可能进一步降低吞吐。该项需要后续在实现阶段检查并节流。

## 目标

- 消除或显著缩短大文件传输开始前停顿。
- 降低断点续传状态维护对吞吐的影响。
- 保持传输正确性和断点续传可靠性。
- 在不破坏现有协议和测试的前提下，先落地低风险优化；协议级优化单独分阶段推进。

## 解决计划

### P0：低风险性能修复

优先级最高，尽量不改变协议语义。

#### 1. 断点续传状态写入节流

修改 `src/transfer/Receiver.cpp`，避免每个 Data 包调用 `resume_manager.update_offset()`。

建议策略：满足任一条件才写入状态：

- 距离上次落盘超过 `64MiB` 或 `256MiB`；
- 距离上次落盘超过 `1s`；
- 文件接收循环结束前强制写入最后 offset；
- 文件完整校验成功后继续执行 `resume_manager.clear(...)`。

预期收益：`5.9GB` 文件的状态文件读写从约 `6000` 次降低到几十次或更少。

#### 2. 进度输出节流

检查 `src/main.cpp` 中的进度输出逻辑。

建议策略：

- 每 `200ms` 或 `500ms` 最多刷新一次；
- 文件完成时强制刷新一次；
- 保留最终完成信息。

预期收益：降低控制台 I/O 对吞吐的干扰，尤其是 Windows 终端环境。

#### 3. 大文件场景 chunk size 建议

当前默认 chunk size 为 `1MiB`。高速局域网大文件传输可先建议使用：

```bash
beamdrop send <file> --to <host:port> --chunk-size 8388608
```

即 `8MiB`。后续可基于实测决定是否调整默认值或按配置推荐。

### P1：消除发送前预哈希停顿

当前 `FILE_INFO` 包含 `sha256`，发送端必须先算完整文件哈希才能发送文件信息。要彻底解决开始前停顿，需要调整校验协议或配置语义。

可选方案：

#### 方案 1：流式哈希 + 文件结束校验包

- `FILE_INFO` 不再强制携带预计算 SHA-256，或允许 `sha256` 为空。
- 发送端边读取文件边更新 SHA-256。
- 接收端边写入边更新 SHA-256。
- 文件数据发送完成后，发送端发送新增包，例如 `FILE_HASH`。
- 接收端用本地流式计算结果和 `FILE_HASH` 比对。

优点：传输可以在读取第一个 chunk 后立即开始。

风险：需要修改协议、编解码、测试和文档；断点续传从非零 offset 开始时，接收端哈希状态处理需要单独设计。

#### 方案 2：配置化关闭强校验

项目配置中已有 `verify_sha256` 字段，但当前传输实现未完整使用该字段控制哈希路径。

可支持：

- `verify_sha256=true`：保持强校验；
- `verify_sha256=false`：跳过发送前预哈希和接收后整文件哈希。

风险：断点续传当前使用 `(relative_path, size, sha256)` 作为身份，关闭哈希后需要替代身份策略，例如 `(relative_path, size, mtime)` 或 `(relative_path, size, empty_hash)`，并明确可靠性边界。

### P2：传输层复制优化

新增面向大 payload 的写入路径，减少 Data 包复制。

候选设计：

- 增加 `write_packet_header_and_payload(connection, header, payload_span)`；
- header 单独序列化并写入；
- payload 直接从文件读缓冲区写入 socket；
- 保留现有 `write_packet()` 用于小包和兼容路径。

预期收益：降低 CPU 和内存带宽开销，提高高速链路下的稳定吞吐。

### P3：性能基准和回归验证

补充可重复的性能验证流程。

建议指标：

- 发送前等待时间；
- 总传输耗时；
- 应用层吞吐；
- 接收端最终校验耗时；
- resume 状态落盘次数；
- 不同 chunk size 下的吞吐对比。

建议场景：

1. 小文件完整传输；
2. `5GB+` 单文件完整传输；
3. 大文件中断后续传；
4. `verify_sha256=true/false` 对比；
5. `1MiB/4MiB/8MiB/16MiB` chunk size 对比。

## 验证标准

### P0 完成标准

- 断点续传状态不再每个 Data 包落盘。
- 中断传输后仍能从最近一次已落盘 offset 续传。
- 完整传输成功后状态记录被清理。
- 现有单元测试和集成测试通过。
- 大文件传输时状态文件写入次数显著下降。

### P1 完成标准

- 大文件发送启动前不再需要完整扫描源文件。
- 开启强校验时仍能发现传输内容不一致。
- 断点续传身份策略明确且有测试覆盖。
- 协议文档同步更新。

### P2 完成标准

- Data 包发送路径减少至少一次大 payload 内存复制。
- 现有协议兼容或迁移路径明确。
- 大文件吞吐有可量化改善。

## 后续执行顺序

1. 先实现 P0：状态落盘节流、进度输出节流、chunk size 建议。
2. 再设计并实现 P1：流式哈希或配置化校验策略。
3. 最后推进 P2：PacketIO 大 payload 零/少复制优化。
4. 每阶段完成后补充测试结果和性能数据到开发日志。
