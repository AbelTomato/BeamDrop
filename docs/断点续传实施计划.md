# 断点续传实施计划

## 1. 目标

实现 BeamDrop MVP 断点续传能力，使传输中断后再次发送同一文件时，接收端能够根据已有文件和状态记录返回可续传 offset，发送端从该 offset 继续发送，最终仍通过文件大小和 SHA256 校验保证结果一致。

## 2. 当前状态

- `PacketType` 已预留 `ResumeReq` 和 `ResumeAck`。
- 配置已包含 `transfer.enable_resume` 和 `transfer.state_file`。
- `Sender` 已在 `FILE_INFO` 后读取 `RESUME_ACK`，并从 ack offset 继续发送。
- `Receiver` 已在 `FILE_INFO` 后回复 `RESUME_ACK`，offset 为 0 时覆盖写入，offset 大于 0 时追加写入。
- 已实现 `ResumeManager`、续传状态文件读写、offset 协商、追加写入和断点续传集成测试。

## 3. 总体约束

- 不修改 Packet Header 格式。
- 不引入第三方 JSON 依赖。
- 不改变现有 `HELLO / FILE_INFO / DATA / FILE_END / FINISH` 会话边界。
- 断点续传必须同时修改发送端和接收端；升级后单文件协议流程统一为 `FILE_INFO -> RESUME_ACK -> DATA -> FILE_END`。
- `RESUME_ACK` 的 offset 必须满足 `0 <= offset <= file_size`。
- 续传状态必须至少绑定 `relative_path + size + sha256`，避免同路径不同内容误续传。
- 如果状态 offset 与实际已存在文件大小冲突，必须选择保守策略，不能跳过未确认写入的数据。
- 每完成一个实现切片，必须立即在 `docs/开发日志.md` 末尾追加一条日志。日志必须包含：实现内容、验证内容、测试结果、新增/修改文件、后续事项。未追加开发日志时，不得进入下一个切片。

## 4. 协议流程

### 4.1 非续传或 offset 为 0

```text
Client -> Server: FILE_INFO(relative_path, size, sha256)
Server -> Client: RESUME_ACK(offset=0)
Client -> Server: DATA...
Client -> Server: FILE_END
```

### 4.2 续传 offset 大于 0

```text
Client -> Server: FILE_INFO(relative_path, size, sha256)
Server 检查 transfer_state.json 和本地已存在文件
Server -> Client: RESUME_ACK(offset=n)
Client 从 offset=n 开始读取源文件
Client -> Server: DATA...
Client -> Server: FILE_END
Server 校验最终 size 和 SHA256
```

## 6. 验收标准

- `ResumeAckCodec` 和 `ResumeManager` 有单元测试覆盖。
- 普通单文件传输、多文件/目录传输仍通过。
- 断点续传集成测试通过。

- 续传完成后最终文件大小与 SHA256 和源文件一致。
- 续传完成后对应状态记录被清理。
- 每个切片完成后均已在 `docs/开发日志.md` 末尾追加日志。

## 7. 暂不纳入本轮范围

- 多客户端并发续传。
- 跨版本协议兼容。
- 每个 `DATA` packet 的独立 checksum。
- 续传状态文件加锁。
- 命令行层面的 `--enable-resume`、`--disable-resume`、`--state-file` 覆盖参数。
- 断点续传过程中的交互式确认或冲突解决 UI。
